Server Smalltalk Manifesto

An object is more than bytes in memory—it's possibility incarnate.

An object can be a promise of future cash flow, waiting to be collected. An object can embody a contractual relationship, evolving with each interaction. An object can represent the delivery of a life-saving drug, carrying with it hope and healing.

But objects only achieve their full potential when they can live, grow, and interact. When they aren't dead, static entries in a database, but living entities in a responsive ecosystem.

Data imprisoned in rows and columns is merely historical. Data embodied in living objects is generative—constantly creating new value through its interactions.

That's what Server Smalltalk makes possible: systems where objects truly live, where value compounds through connection, where the digital mirrors the organic vitality of the world itself.

And beneath it all: the same old goal. Encouraging the creative spirit in everyone.

Principles

Large Memory is a Feature, Not a Problem

Most systems pretend memory is scarce. Ours assumes it's cheap. If you can afford to remember, you can afford to think more clearly. Objects persist in RAM not because it's trendy, but because it's humane—navigable, inspectable, alive.

Persistence is the Default

In traditional systems, persistence is something you earn. In Server Smalltalk, it's the baseline. If you don't say otherwise, your data survives. The system encourages habits of care, not hacks of storage.

Transactions Define Intent

A transaction isn't a scary word from the database world. It's a tool for saying: "This goes together." Transactions in Server Smalltalk help systems grow in coherent, recoverable steps. They're optimistic, rollback-friendly, and integrated.

Parallelism Shouldn't Hurt

Concurrency is the price of scale, but it doesn't have to be the price of sanity. Server Smalltalk favors message passing over locks, actors over race conditions, clarity over cleverness. You don't need a PhD to write safe, parallel code.

The System is the Source

There's no distinction between "source" and "running system." The image is the truth. Code lives where it runs, changes where it's needed, and grows in the same place users feel its effects. Version control exists, but it's in service to the live world, not the other way around.

Everything is Live

Debuggers open on the server. Logs are objects. Stack traces are clickable. A crash is not the end—it's an invitation to learn, adjust, retry. You're never blind, never shut out. Development continues at runtime.

Interop is Table Stakes

Smalltalk doesn't live in a vacuum. Server Smalltalk speaks HTTP, talks to databases, runs scheduled jobs, listens to queues. It integrates because that's what creative systems do—they converse.

Final Note

Server Smalltalk isn't just infrastructure. It's an invitation. To build things that live. To maintain systems that stay up and stay kind to the people working on them. To close the gap between idea and reality.

And most of all, to encourage the creative spirit in everyone who touches the system—developer, ops, or user.

Join the movement. Build systems that live.
